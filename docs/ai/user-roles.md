# User Roles and Permissions

Этот документ описывает ролевую систему (RBAC), которую должен учитывать агент при разработке новых фичей или написании API.

## Доступные роли в системе (Enum UserRole)

| Роль                   | Описание                                                                                                                                                                                                            |
| :--------------------- | :------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **ATHLETE**            | Обычный пользователь. Может отслеживать свои тренировки, просматривать свои результаты, участвовать в командных событиях (если добавлен в команду).                                                                 |
| **TRAINER**            | Тренер. Может создавать и управлять командами (как `Owner`), добавлять атлетов по инвайт-кодам, планировать и создавать командные тренировки (Events).                                                              |
| **ORGANIZATION_ADMIN** | Администратор организации. Управляет всей организацией, включая множество команд и тренеров внутри своей организации. Имеет доступ к статистике организации.                                                        |
| **SUPER_ADMIN**        | Системный администратор. Имеет полный доступ ко всем ресурсам системы: от управления пользователями (бан/разбан, смена ролей) до редактирования глобальных тренировок и контента. Исключительный доступ к `/admin`. |

## Матрица доступов (Ориентир для Guard-ов)

При создании контроллеров NestJS используйте `@Roles(...)` в соответствии с этой матрицей таблицей:

- Просмотр и редактирование своего профиля: Все роли.
- Создание команды (`TeamsService.create`): `TRAINER`, `ORGANIZATION_ADMIN`, `SUPER_ADMIN`.
- Редактирование/Удаление команды: Создатель команды (обычно `TRAINER` со статусом Owner), `ORGANIZATION_ADMIN`, `SUPER_ADMIN`.
- Управление участниками команды: `TRAINER` (в своей команде), `ORGANIZATION_ADMIN`, `SUPER_ADMIN`.
- Создание мероприятий (`Events`): `TRAINER` (для команды), `ORGANIZATION_ADMIN`, `SUPER_ADMIN`. Так же, `ATHLETE` может создавать персональные события только для себя.
- Управление контентом платформы (Упражнения, Глобальные WOD, Новости, Контентные блоки): **Только** `SUPER_ADMIN` (иногда делегируется другим ролям, но базовая проверка должна быть строгой).
- Управление чужими аккаунтами (Смена роли, бан `UsersController`): **Только** `SUPER_ADMIN`.

## На что обращать внимание

1. **Защита чужих данных:** Атлет (ATHLETE) не должен иметь возможность запрашивать, изменять или удалять чужие `EventResult`, `User` профиль или `StrengthWorkoutResult`, если это не предусмотрено логикой (например, публичные профили). Всегда сверяйте `req.user.id` с ID ресурса.
2. **Командные права:** Для взаимодействия с командой (например, добавление результата за команду или чтение закрытых командных событий) нужно проверять наличие пользователя в таблице `team_members` с соответствующим `teamId` и ролью.
