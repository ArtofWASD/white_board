# API Integration & Data Models Guide

В этом документе собраны правила интеграции фронтенда и бэкенда, а также детали реализации конкретных сущностей на основе существующих эндпоинтов.

## 1. Разработка новых API Endpoints

При написании новых роутов на бэкенде, строго следуйте устоявшейся структуре ответов:

- **Успешный ответ (200/201):** Должен возвращать либо чистый массив объектов (для списков), либо JSON-объект, где данные лежат прямо в корне (например, возвращается сам объект ресурса или объект через вложенный ключ вроде `{ user: {...}, token: "..." }` для авторизации).
- **Ошибки (4xx/500):** Всегда возвращают объект с полем `message`: `{ "message": "Описание ошибки" }`. Бэкенд реализован с применением стандартных NestJS Exceptions (`BadRequestException`, `NotFoundException`, `InternalServerErrorException`), которые сами формируют правильный JSON.

## 2. Сущности и их особенности

### Тренировки (Events)

События (`Event`) разделяются на два типа статусов:

- `FUTURE` — запланированная тренировка.
- `COMPLETED` / `PAST` — прошедшая тренировка.
  У события обязательно есть `eventDate`, `scheme` (`FOR_TIME`, `AMRAP`, `EMOM` и др.) и владелец `userId`. Событие может быть привязано к команде `teamId`.

### Результаты тренировок (EventResult)

- Привязаны к конкретному `Event` через `eventId`.
- Содержат поля `time` (строка, например "10:30"), `value` (число, например количество повторений), `scaling` (модификатор: "RX", "SCALED").
- На бэкенде контроллеры для результатов часто вкладываются в путь родителя: `POST /events/:eventId/results`.

### Команды (Teams)

- Команда имеет `ownerId` (тренер, создавший ее).
- Связь пользователей с командой осуществляется через промежуточную таблицу `TeamMember`, где указывается роль юзера внутри команды (`MEMBER`, `ADMIN`, `OWNER`).
- Для приглашения атлетов в команду используется уникальный `inviteCode`. Эндпойнт инвайта должен проверять валидность кода и добавлять связь в `TeamMember`.

## 3. Интеграция с Frontend (Связки)

При написании фронтенд вызовов API:

1. Не ходите напрямую через `fetch` в UI-компонентах. Всю логику общения с API необходимо выносить в специальные модули внутри `frontend/src/lib/api/` (например `authApi`, `eventsApi`, `teamApi`).
2. Эти API модули затем вызываются либо из Zustand сторов для глобальных состояний, либо через React Server Components / Client fetcher-ы в связке с хуками загрузки для локальных данных.
3. Обязательно подставляйте авторизационный токен (обычно `Bearer JWT`) в заголовки запросов, если API-клиент не настроен делать это автоматически с помощью axios/interceptor-ов.
