# Architecture Standards & Rules

В этом документе описаны основные архитектурные стандарты, которых должен придерживаться агент при написании кода для проекта Whiteboard.

## 1. Backend: NestJS API & Controllers

- **Структура:** Бизнес-логика должна находиться в `services`, а обработка HTTP-запросов — в `controllers`. Не пишите сложную логику проверки прав или расчетов прямо в контроллерах.
- **Роутинг и Декораторы:**
  - Используйте стандартные REST методы: `@Get()`, `@Post()`, `@Patch()`, `@Delete()`.
  - Для валидации UUID в параметрах строго используйте `@Param('id', ParseUUIDPipe)`.
- **Авторизация и Права доступа:**
  - Роуты, требующие авторизации, должны быть защищены `@UseGuards(JwtAuthGuard)`.
  - Для проверки ролей используйте комбинацию `@UseGuards(JwtAuthGuard, RolesGuard)` и кастомный декоратор `@Roles(UserRole.ATHLETE, ...)` с передачей допустимых ролей из Enum Prisma.
- **DTO:** Все входящие данные в контроллеры (`@Body()`) валидируются с помощью классов DTO (с декораторами `class-validator`).
- **Real-time и фоновые задачи (Cron):**
  - Подключения по WebSocket (Socket.IO) обрабатываются в соответствующих Gateway-классах (отдельно от `controllers`).
  - Фоновые планировщики задач реализованы через пакет `@nestjs/schedule`.

## 2. Backend: Работа с базой данных (Prisma)

- **Файл схемы:** `backend/prisma/schema.prisma`.
- **Именование таблиц и колонок:**
  - В Prisma Client (TypeScript) мы всегда используем `camelCase` (например, `createdAt`, `userId`).
  - В самой PostgreSQL названия создаются в `snake_case`. Для этого в схеме всегда используем атрибут `@map("..._...")`. Пример: `createdAt DateTime @map("created_at")`.
  - Названия таблиц задаются во множественном числе в `snake_case` через `@@map("table_names")`.
- **Первичные ключи:** Для всех моделей используем UUID: `id String @id @default(uuid())`.
- **Связи:**
  - Реляции детально прописываются с явным указанием `fields` и `references`.
  - Внимательно относитесь к удалению связанных данных: используйте `onDelete: Cascade` там, где это целесообразно (например, удаление результатов при удалении пользователя/события).

## 3. Frontend: Стейт-менеджмент (Zustand)

- **Расположение:** Файлы сторов лежат в `frontend/src/lib/store/` (например, `useAuthStore.ts`).
- **Синтаксис и типизация:**
  - Используйте `create<IStoreType>()(...)` для строгой типизации всего состояния и экшенов.
- **Взаимодействие с API:**
  - Не пишите вызовы `fetch`/`axios` напрямую в компонентах (за редким исключением).
  - Сторы Zustand должны дёргать API-клиент (вынесенный отдельно, например `authApi`) и сохранять результат в глобальное состояние.
- **Персистентность:**
  - Если данные нужно сохранить между перезагрузками страницы, используйте middleware `persist` поверх `create`.
  - Хранилищем выступает `createJSONStorage(() => localStorage)`. Используйте `partialize`, чтобы сохранять только нужные поля (например, пользователя, но без конфиденциальных токенов, если они лежат в httpOnly куках или память-only).

## 4. Frontend: Формы и Валидация (react-hook-form + zod)

- **Построение форм:** Для создания форм всегда используйте `react-hook-form`. Мы стараемся избегать неконтролируемых компонентов с кучей `useState` для каждого текстового поля.
- **Валидация:** Используйте строгую типизацию и валидацию через библиотеку `zod`. Схемы валидации (например, `const loginSchema = z.object({...})`) должны находиться в файле с компонентом или быть вынесены в отдельные файлы `/frontend/src/lib/validations`.
- **Интеграция:** Для связки `react-hook-form` и `zod` используйте `@hookform/resolvers/zod`. Обработка сообщений об ошибках должна происходить внутри UI-компонент (как правило, с применением компонента ошибки или подсказки).

## 5. Общие принципы

- **Типизация:** Максимально строгая типизация TypeScript. Избегайте `any` и `ts-ignore`.
- **Импорты (Frontend):** Используйте абсолютные импорты (`@/...`) там, где это настроено в проекте.
- **Чистота кода:** Придерживайтесь принципов SOLID. Модули должны быть независимыми и переиспользуемыми.
